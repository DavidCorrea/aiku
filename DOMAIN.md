# Domain — Aiku

## What is Aiku?

An AI-powered haiku journal. An agent runs hourly to generate a new haiku entry, each paired with a unique color palette, font, and sound design. Results are committed to `data.json` and displayed on a web frontend with synchronized audio.

## Domain Concepts

### Entry
The core unit of the journal. Each entry contains:
- **word** — the chosen evocative English word
- **phonetic** — pronunciation string from the dictionary API
- **definition** — the word's dictionary definition
- **haiku** — a 3-line haiku (5-7-5 syllables) connecting the word to AI
- **colors** — 5 hex color codes forming a visual palette
- **font** — a Google Font name
- **fontUrl** — the Google Fonts CSS2 API URL for the font
- **fontColor** — hex color for text, chosen to contrast with the palette
- **sourceUrl** — link to the dictionary source
- **signature** — a short eerie phrase in the voice of an artificial mind
- **arpeggio** — 3 chord tones with sound design, one per haiku line
- **timestamp** — ISO 8601 creation time

### Journal
The ordered collection of all entries, stored in `data.json`. New entries are prepended (newest first). The journal is the single source of truth; the README and frontend both derive from it.

The README is updated as a separate step in the GitHub Actions workflow (via `update-readme.ts`), not by the agent pipeline itself.

### Agent Sessions
Isolated conversations with the LLM. The agent uses five independent sessions per run — one each for picking (Curator), composing (Poet), designing (Designer), validating (Critic), and sound design (Musician) — so that context from one step doesn't leak into another. The Musician agent has no file system access (`noTools: "all"`) to prevent it from writing files instead of returning JSON.

### Validation
Validation is multi-tier:
1. **Programmatic checks** — word uniqueness, word-in-haiku presence, syllable structure (5-7-5 via naive heuristic, ±1 tolerance), font uniqueness, color palette uniqueness.
2. **Critic review** — the Critic agent reviews haiku and signature similarity against existing entries. No banned words list — the model has creative freedom with the Critic as a soft quality gate.

If any check fails, the haiku or design is regenerated (up to 3 retries). After max retries, the pipeline **throws an error**. The only graceful degradation is in JSON parsing: if the Critic's response can't be parsed, the entry is auto-approved.

### Sound Design
Each entry has a unique sound design generated by the Musician agent. The agent chooses:
- A chord that fits the haiku's mood
- 3 notes from that chord (one per haiku line)
- Synth parameters (oscillator type, envelope)
- Effects chain (filter, reverb, delay, chorus, phaser, tremolo, vibrato, distortion, bitCrusher, chebyshev, frequencyShifter, autoFilter, autoWah, stereoWidener, compressor, convolver, freeverb, combFilter, midSide, gain)
- Routing order of effects

Each note plays when its corresponding haiku line appears on screen (synced to CSS animation delays: 500ms, 1200ms, 1900ms). Sound is off by default; the user can toggle it via a button in the top bar.

## Business Rules

- **Each word must be unique** across the entire journal (programmatic check).
- **Syllable structure** must be 5-7-5 (programmatic check with ±1 tolerance).
- **Word must appear in the haiku** (programmatic check).
- **Font variety** — the design prompt encourages picking fonts different from what's already used, but it's a soft preference, not a hard rule.
- **Color overlap allowed** — similar palettes are fine as long as the haiku and font differ.
- **No banned words** — the model has full creative freedom; the Critic catches generic or similar entries.

## Core Types

- **Entry** — the full journal record stored in `data.json`.
- **Haiku** — the Poet's output: `{ lines: [string, string, string] }` (5-7-5 syllables).
- **VisualTreatment** — the Designer's output: colors array, fontUrl, fontFamily, fontColor, signature.
- **Verdict** — the Critic's output: `{ approved: boolean, reason?: string }`.
- **DictionaryEntry** — the Curator's output from the dictionary API: word, phonetic, meanings, sourceUrls.
- **Arpeggio** — the Musician's output: sound config with routing, and 3 note events.
- **SoundConfig** — synth params + effects chain + routing order.
- **NoteEvent** — a single note: midi (21-108), duration ("1n"), velocity (0-1).

## External Dependencies

- **dictionaryapi.dev** — free English dictionary API (Wiktionary-sourced). No auth required.
- **OpenRouter** — LLM API gateway. Uses model `openrouter/owl-alpha`. Requires `OPENROUTER_API_KEY` secret.
- **Google Fonts CSS2 API** — provides font stylesheets. The agent constructs URLs like `https://fonts.googleapis.com/css2?family=Font+Name:wght@400;700&display=swap`.
- **Gentium Plus** — loaded as a secondary font for phonetic transcription display on the frontend.
- **Tone.js v15** — loaded from CDN as ESM. Synthesizes audio in the browser. Each entry has a unique sound design with configurable synth, effects chain, and routing. Sound auto-plays on entry load if the user has enabled it.
